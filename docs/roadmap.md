# Roadmap

`GalaxyOPC` 的开源版本有两个目标：

1. 在 GitHub 上形成公开、可信、可协作的技术名片。
2. 为面向非技术用户的本地运行版本提供代码底座。

因此后续迭代不会只围绕“源码可运行”，而是会重点解决“不会技术的人能不能装、能不能配、能不能马上用”的问题。

## Phase 1: 本地运行版本

目标：让用户不必理解 Node.js、数据库、环境变量，也能在本地跑起 OPC。

计划：

- 封装 Windows 安装包。
- 封装 macOS 安装包。
- 内置 Node.js 运行时和本地 SQLite。
- 默认启用 `LOCAL_MODE=true`。
- 首次启动自动创建本地数据库和管理员账号。
- 提供一键打开本地 Web 控制台。

现有基础：

- 已支持 SQLite 本地模式。
- 已存在 `LOCAL_MODE` 和本地代理能力。
- 原项目中已有 Electron / Tauri 桌面版工程，可作为封装基础。

## Phase 2: 首次配置向导

目标：让非技术用户像配置普通软件一样完成模型、搜索、邮箱和协作通道设置。

计划：

- 首次启动展示配置向导。
- 配置项按“必填 / 可选 / 稍后配置”分组。
- 模型配置支持“星环免费试用通道”和“自配 API Key”两种路径。
- 搜索、邮箱、飞书等能力提供状态检测。
- 配置完成后直接进入 AI 对话入口。

推荐交互：

- 第一步：选择使用方式。
- 第二步：选择模型能力。
- 第三步：配置协作通道。
- 第四步：创建第一个公司或工作空间。
- 第五步：进入 AI 对话。

## Phase 3: 默认可用的星环算力通道

目标：降低 API Key 配置门槛，让用户先体验再决定是否深入配置。

计划：

- 默认提供有限额度的星环试用算力。
- 在本地版模型配置页显示当前模式：试用通道 / 自配模型 / 混合模式。
- 引导用户在额度不足或生产部署时切换到自己的模型服务。
- 保留完全自托管能力，避免平台绑定。

注意：

- 试用通道应有额度、频率和风控限制。
- 文档中需要明确数据流向和隐私边界。

## Phase 4: 协作通道打通

目标：让 OPC 不只是网页后台，而是进入用户真实工作的聊天与协同入口。

优先级：

1. 飞书：已有长连接与工具集基础，优先打磨到可产品化。
2. 企业微信：补齐机器人、通讯录、消息、群聊和任务场景。
3. 钉钉：补齐机器人、待办、日程和审批场景。

关键场景：

- 用户在飞书 / 企业微信 / 钉钉里直接问 AI。
- AI 自动生成会议纪要、周报、客户跟进和项目任务。
- OPC 中的项目、待办、文档可以同步到协作平台。
- 协作平台中的消息可以沉淀回 OPC 记忆与业务上下文。

## Phase 5: 空间页面重构

目标：重新划分产品信息架构，让不同入口承载不同世界。

建议分工：

- AI 对话：作为智能入口，负责自然语言调度。
- 管理后台：作为软件系统，负责账号、公司、项目、订单、配置等管理。
- 城市管理：作为物理世界的城市与节点空间，独立成横向大页面。
- 物联网管理：作为物理空间与设备的数字孪生页面，独立成横向大页面。

页面方向：

- 城市管理不再只是后台菜单，而是城市节点、OPC 中心和资源地图。
- 物联网管理不再只是功能卡片，而是空间、楼层、房间、设备、状态的可视化界面。
- AI 对话可以调起城市和物联网页面，但不替代它们的独立空间表达。

## 衡量标准

一轮迭代是否有效，不只看代码功能是否存在，而看以下问题：

- 非技术用户能否在 10 分钟内装好并进入对话？
- 用户是否不配置 API Key 也能先体验核心能力？
- 飞书 / 企业微信 / 钉钉是否能成为真实入口，而不是配置摆设？
- 城市和物联网页面是否能让用户明显感受到“物理世界被数字化”？
- 开源仓库是否持续保持清晰、可信、可二次开发？
